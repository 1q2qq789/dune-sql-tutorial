Section 01

EVM 与数据表的映射关系

EVM（以太坊虚拟机）是以太坊、Polygon、Arbitrum、Optimism 等链共用的标准，因此它们的原始数据输出格式一致。理解一笔交易如何产生多张数据表的数据，是一切查询的出发点。

💡
核心记忆点： 所有表都通过 tx_hash（交易哈希）相互关联。一笔交易可以在多张表中产生多行数据，但在 transactions 表中始终只有一行。

一笔 Uniswap Swap 会写入哪些表？

表名写入内容行数
ethereum.transactions整笔交易记录（from/to/input/gas）1 行
ethereum.logsSwap() 事件 + 2个 Transfer() 事件3 行
ethereum.tracesswap() 函数调用 + 2个 transfer() 调用3+ 行
uniswap_v2_ethereum.Pair_evt_Swap解码后的 Swap 事件1 行
erc20_ethereum.evt_Transfer代币转入/转出事件（Spellbook）2 行
dex.trades聚合交易数据（Spellbook）1 行
⚠️
关键区别： tx_hash 在 transactions 表里是唯一的，但在 logs / traces 等表里可能有多行（因为一笔交易可触发多个事件/调用）。

Section 02

Dune 表的五个层级

🟡 Raw 原始表

transactions logs traces blocks。数据最原始，大量字节数组（0x开头），实时性最强，但查询最难。

🔵 Decoded 解码表

基于合约 ABI 解码，如 uniswap_v2_ethereum.Pair_evt_Swap。格式：namespace_chain.Contract_evt_EventName。

🟢 Spellbook 魔法书

用 SQL 聚合构建的高层表，如 dex.trades erc20_ethereum.evt_Transfer tokens.erc20。有数据延迟。

🟠 Community 社区数据

第三方数据集，如 Flashbots MEV 数据、Reservoir NFT 数据，通过 tx_hash 与 Dune 数据关联。

🟣 User Views 用户视图

每个公开查询都变成视图，用 query_<id> 语法引用。只支持 DuneSQL 引擎的查询。

✅
选表原则：优先用 Spellbook（最省力）→ Decoded（精确查某合约）→ Raw（需要原始字节数据时）。
代币相关表始终用 地址（address）而非 symbol 做关联，避免同名代币混淆。

常用 Spellbook 表速查

表名用途
tokens.erc20ERC20 代币名称、小数位数（必备 JOIN 表）
erc20_ethereum.evt_Transfer以太坊全部 ERC20 转账事件
prices.usd代币 USD 价格（API来源）
dex.prices代币价格（DEX 汇率计算）
dex.trades所有 DEX 聚合交易数据

Section 03

基础 SQL 语法

DuneSQL 基于 Trino SQL，以下是查询的完整骨架结构（方括号为可选）：

SQL 查询骨架
-- 1. 可选：定义公共表表达式 (CTE)
[WITH subquery_name AS (...)]

-- 2. 核心：选择列
SELECT [DISTINCT] col1, col2, COUNT(*) AS cnt

-- 3. 数据来源
FROM some_table

-- 4. 可选：表关联
[LEFT JOIN other_table ON some_table.tx_hash = other_table.tx_hash]

-- 5. 过滤条件（在聚合前）
[WHERE block_time >= NOW() - INTERVAL '7' DAY]

-- 6. 分组
[GROUP BY col1, col2]

-- 7. 过滤条件（在聚合后）
[HAVING COUNT(*) > 10]

-- 8. 排序 & 分页
[ORDER BY cnt DESC]
[LIMIT 100]

示例：查询 Aave v3 在 Optimism 上的唯一供应者

SELECT DISTINCT
  user AS supplier_address
FROM aave_v3_optimism.Pool_evt_Supply
WHERE evt_block_time >= NOW() - INTERVAL '30' DAY

UNION ALL：合并多个 Uniswap v2 swap 函数

-- Uniswap v2 Router 有多个 swap 函数，需要 UNION ALL 全部取到
WITH all_swaps AS (
  SELECT evt_tx_hash, evt_block_time, 'swapExactTokensForTokens' AS fn_name
  FROM uniswap_v2_ethereum.Router_call_swapExactTokensForTokens

  UNION ALL

  SELECT evt_tx_hash, evt_block_time, 'swapExactETHForTokens'
  FROM uniswap_v2_ethereum.Router_call_swapExactETHForTokens
)
SELECT DATE_TRUNC('day', evt_block_time) AS day,
       COUNT(*) AS total_swaps
FROM all_swaps
GROUP BY 1
ORDER BY 1 DESC

Section 04

数据类型与 CAST 转换

链上数据字段经常以字符串（VARCHAR）或大整数（BIGINT）存储，必须正确转换才能计算。

类型说明常见场景
VARCHAR字符串，大多数列的默认类型地址、符号、名称
DOUBLE双精度浮点数，用于所有数值计算金额、价格、比率
VARBINARY字节数组，0x开头的原始数据data、topics、input 列
TIMESTAMP时间戳，时间函数的基础类型block_time、evt_block_time
BIGINT大整数，代币原始金额value、amount（需除以精度）

CAST 的四种核心用法

-- ① 字符串 → 数值（字段存成 '1500.50' 的情况）
CAST(amount_usd AS DOUBLE) AS amount

-- ② 整数除法（不转换 1/2=0，转换后 1/2=0.5）
CAST(1 AS DOUBLE) / 2  -- 正确返回 0.5

-- ③ 字符串 → 时间戳（用于时间过滤）
CAST('2026-01-01' AS TIMESTAMP)

-- ④ UNION 拼表时统一列类型
SELECT CAST(amount AS DOUBLE) FROM table_a
UNION ALL
SELECT CAST(volume AS DOUBLE) FROM table_b

-- ⑤ 实战：ERC20 交易量统计（需先转类型再 SUM）
SUM(CAST(amount_usd AS DOUBLE)) AS usd_volume
⚠️
常见报错原因：对超大整数（如代币原始金额 1e18 量级）直接 SUM，超出 BIGINT 上限会报错崩溃。务必先 CAST AS DOUBLE 再聚合。

代币金额的精度处理

-- 原始 value 是 wei (1 ETH = 1e18 wei)，需要除以 10^decimals
SUM(CAST(value AS DOUBLE) / POW(10, t.decimals)) AS amount_token

-- 如果 JOIN tokens.erc20 失败（没有该代币），用默认 18 位精度
SUM(CAST(value AS DOUBLE) / COALESCE(POW(10, t.decimals), 1e18)) AS amount_token

Section 05

时间函数 · 聚合 · JOIN

三个每天必用的时间函数

-- 1. date_trunc：向下取整到指定精度（类似 floor）
DATE_TRUNC('day', block_time)     -- 2026-03-11 14:32 → 2026-03-11 00:00
DATE_TRUNC('month', block_time)   -- → 2026-03-01 00:00

-- 2. NOW()：查询运行时的当前 UTC 时间
WHERE block_time >= NOW() - INTERVAL '7' DAY

-- 3. 时间字符串转 timestamp
WHERE block_time > TIMESTAMP '2026-01-01 00:00:00'

常用聚合函数

COUNT(*)                          -- 总行数
COUNT(DISTINCT sender)           -- 唯一用户数
SUM(CAST(amount AS DOUBLE))      -- 总金额
AVG(gas_used)                     -- 平均 gas
MAX(block_time)                   -- 最新时间
APPROX_DISTINCT(sender)          -- 近似唯一数（更快）
APPROX_PERCENTILE(amount, 0.5)  -- 中位数

-- HAVING：GROUP BY 之后的过滤（不是 WHERE！）
GROUP BY sender
HAVING SUM(CAST(amount_eth AS DOUBLE)) > 100
-- 筛选出过去 10 天内累计交换超过 100 ETH 的地址

JOIN 使用场景

-- 场景 1：LEFT JOIN tokens.erc20 获取代币符号和精度
SELECT
  t.contract_address,
  COALESCE(tok.symbol, 'Unknown') AS symbol,
  SUM(CAST(t.value AS DOUBLE) / POW(10, COALESCE(tok.decimals, 18))) AS amount
FROM erc20_ethereum.evt_Transfer t
LEFT JOIN tokens.erc20 tok
  ON t.contract_address = tok.contract_address
  AND tok.blockchain = 'ethereum'
WHERE t.evt_block_time >= NOW() - INTERVAL '1' DAY
GROUP BY 1, 2
ORDER BY amount DESC
LIMIT 50

-- 场景 2：JOIN transactions 获取 from/to（事件表本身没有这些列）
SELECT
  s.evt_tx_hash,
  tx.from AS wallet_address
FROM uniswap_v2_ethereum.Pair_evt_Swap s
JOIN ethereum.transactions tx
  ON s.evt_tx_hash = tx.hash
WHERE s.contract_address = 0xB4e16d0168e52d35CaCD2c6185b44281Ec28C9Dc
⚠️
不要盲信列名！ 事件表里的 sender ≠ 交易的 from。务必搞清楚每列的实际含义，必要时 JOIN transactions 表获取真实来源地址。

Section 06

容错处理：防崩溃三件套

链上数据常有缺失值、除以零、类型转换失败等异常情况，以下三个函数是防崩溃必备。

① NULLIF — 防止除以零报错

-- 原理：如果 dex_volume = 0，则返回 NULL（任何数 / NULL = NULL，不会报错）
a.agg_volume / NULLIF(d.dex_volume, 0) * 100 AS aggregator_pct

-- 完整解读：
-- NULLIF(d.dex_volume, 0) → 若值为0，替换为 NULL
-- agg_volume / NULL        → 结果是 NULL（不报错，不崩溃）
-- * 100 AS aggregator_pct  → 格式化为百分比展示

② COALESCE — 空值兜底替换

-- 返回第一个非 NULL 的值
SELECT COALESCE(NULL, NULL, 1)    -- 返回 1
SELECT COALESCE(NULL, 2, NULL, 32) -- 返回 2

-- 实战：LEFT JOIN 失败时用默认精度
COALESCE(tok.decimals, 18) AS decimals

-- 实战：代币符号找不到时显示 Unknown
COALESCE(tok.symbol, 'Unknown') AS symbol

-- 实战：组合 try() + coalesce() 设置兜底值
SELECT COALESCE(TRY(1/0), 10)  -- 返回 10

③ TRY() — 静默处理运行时错误

SELECT 1/0               -- ❌ 报错，查询终止
SELECT TRY(1/0)          -- ✅ 返回 NULL，查询继续
SELECT COALESCE(TRY(1/0), 10)  -- ✅ 返回 10

-- 适合用于：类型转换可能失败、除法可能出现0分母的场合
💡
实战记忆法：
• NULLIF(col, 0) → 除法分母用它，防除零
• COALESCE(col, default) → JOIN 之后用它，防 NULL
• TRY(expr) → 不确定能否成功的计算用它，防崩溃

Section 07

增量查询模式（Incremental Query）

对于需要每天刷新的时序数据，全量扫描成本极高。增量查询模式通过「只扫描新增数据」的方式大幅降低成本。

💡
核心逻辑： 首次运行扫全量，之后每次只扫 cutoff 之后的新数据，再与历史结果 UNION ALL 合并输出。

完整增量查询模板（以太坊每日交易统计）

-- ① 加载上一次运行的历史结果（首次运行为空表）
WITH prev AS (
  SELECT * FROM TABLE(previous.query.result(
    schema => DESCRIPTOR(
      day        TIMESTAMP(3),
      tx_count   BIGINT,
      total_gas  BIGINT
    )
  ))
),

-- ② 找到"截止点"：历史最新一天往前退一天（lookback window）
checkpoint AS (
  SELECT
    COALESCE(MAX(day), TIMESTAMP '2015-07-30')
      - INTERVAL '1' DAY AS cutoff
  FROM prev
)

-- ③ 保留历史旧数据（不需要重新计算的部分）
SELECT day, tx_count, total_gas
FROM prev
WHERE day < (SELECT cutoff FROM checkpoint)

UNION ALL

-- ④ 计算 cutoff 之后的新数据
SELECT
  DATE_TRUNC('day', block_time) AS day,
  COUNT(*)                       AS tx_count,
  SUM(gas_used)                   AS total_gas
FROM ethereum.transactions
WHERE block_time >= (SELECT cutoff FROM checkpoint)
GROUP BY 1

-- ⑤ 合并历史 + 新增，按时间排序
ORDER BY day

图解增量逻辑

历史结果:   2018-01-01  ...  2025-01-05
                                    ↓
cutoff    = MAX(day) - 1 day = 2025-01-04
                                    ↓
保留旧数据:  2018-01-01  ...  2025-01-03  (day < cutoff)
重新计算:   2025-01-04  ...  今天         (block_time >= cutoff)
                                    ↓
最终输出:   UNION ALL 拼接 → 完整时间序列
✅ 优势

首次运行全量扫描后，后续每次只扫新增区间，大幅减少计算成本（Dune 按量计费）

⚠️ 注意

往前退一天是 lookback window，避免最后一天数据不完整导致漏统计。设计时要考虑这个边界。
